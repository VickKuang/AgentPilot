# 模块进度文档（MVP）

> 更新时间：2026-04-28
> 
> 说明：本进度基于当前仓库实现状态梳理，按“已实现 / 未实现（或仅占位）/ 建议下一步”分项列出。

## 1. 任务管理模块

**进度：90%（核心已完成）**

### 已实现
- 任务创建、查询列表、查询详情 API。
- 任务补参（支持 JSON merge）与必填参数缺失检测。
- 敏感字段自动脱敏（`password/token/secret`）。
- 任务状态流转：`pending/running/paused/passed/failed/blocked/terminated`。

### 未实现 / 待完善
- 任务状态变更约束较弱（例如某些非法状态迁移未统一校验）。
- 缺少分页、过滤、排序查询能力（任务数量大时不友好）。

### 建议下一步
- 增加状态机校验规则（统一迁移矩阵）。
- 为任务列表增加分页与状态过滤参数。

---

## 2. 执行编排（Planner → Observer → Action → Verifier）模块

**进度：80%（流程打通，执行器为 mock）**

### 已实现
- 基于用户目标的场景规划器：登录、搜索、表单、筛选、异常提示、通用场景。
- 执行管线支持：启动、重试、暂停、恢复、终止。
- 支持步骤级重试、全局超时与步骤超时。
- 执行过程中可写入步骤日志、工具调用日志、页面快照。

### 未实现 / 待完善
- Observer/Action 当前为 mock（页面结构、截图 URL、工具输出均为模拟值）。
- Verifier 规则较简单，主要做字符串与元素名检查，未接入真实断言引擎。
- 暂停/恢复为状态级控制，未实现精细化断点续跑（当前由 pipeline 重新拉起循环）。

### 建议下一步
- 接入 Playwright/WebDriver/Appium 真实执行层。
- 引入可配置断言 DSL（文本、属性、可见性、网络响应）。
- 为恢复执行增加“上次成功 step 后续继续”的明确语义。

---

## 3. 可观测性模块（日志/快照/进度）

**进度：85%（可用）**

### 已实现
- 步骤日志查询 API。
- 工具调用日志查询 API（含请求/响应/耗时）。
- 页面快照查询 API。
- 执行进度 API（总步骤、完成数、成功/失败数、百分比）。

### 未实现 / 待完善
- 未提供日志检索能力（按 step、时间、状态筛选）。
- 快照仅记录 mock 路径，缺少真实静态资源落盘与可访问链接。

### 建议下一步
- 增加日志筛选参数与失败聚合视图。
- 对接对象存储并输出可打开的截图 URL。

---

## 4. 报告模块（测试报告 / Bug 报告 / Markdown 导出）

**进度：88%（核心闭环已完成）**

### 已实现
- 测试报告 API。
- Bug 报告 API（失败时自动生成严重级别与复现步骤）。
- Markdown 报告导出 API。
- 报告汇总包括：结果、摘要、步骤、截图列表等。

### 未实现 / 待完善
- Bug 严重级别策略较静态（仅按场景粗分 P1/P2）。
- Markdown 模板较简化，尚未支持自定义模板与多格式导出（PDF/HTML）。

### 建议下一步
- 引入可配置的严重级别评估规则。
- 支持报告模板化与多格式导出。

---

## 5. 前端页面模块（index/task/report）

**进度：75%（演示可用）**

### 已实现
- 任务创建页、执行页、报告页三页打通。
- 执行页支持控制动作与 1.5s 自动刷新。
- 报告页支持查看报告与导出 Markdown。

### 未实现 / 待完善
- 缺少输入校验与错误提示体验（例如 JSON 解析异常未友好提示）。
- 缺少权限体系、用户体系、多项目隔离。
- 缺少实时推送（当前通过轮询实现）。

### 建议下一步
- 增加前端错误处理和表单校验。
- 升级为 WebSocket 实时推送进度。

---

## 6. 存储与数据层模块

**进度：100%（模块目标已落地）**

### 已实现
- 存储层已抽象为可切换实现：默认本地 `data/store.json`，设置 `DATABASE_URL` 后自动切换 PostgreSQL。
- PostgreSQL 持久化已在后端真实接入，支持启动加载与运行期持续写入。
- 写入路径引入事务边界与乐观并发控制（版本号 + 冲突重试）。
- 新增审计字段（`created_at` / `updated_at`）用于存储状态追踪。

### 未实现 / 待完善
- 目前 PostgreSQL 为“状态快照表”模式，后续可按业务表做更细粒度拆分（任务/步骤/日志分表读写）。
- 跨实例高并发场景建议补充压测与监控告警阈值。

### 建议下一步
- 逐步将快照式存储演进为领域 Repository（按任务、日志、报告拆分）。
- 为历史数据引入归档策略（冷热分层 + 对象存储）。

## 总体结论（当前版本定位）

- 当前代码已达到 **“MVP 演示闭环”**：能创建任务、执行模拟流程、查看日志与报告。
- 若目标是 **生产可用自动化测试平台**，主要缺口集中在：
  1) 真实执行引擎，2) 生产级存储，3) 安全与多租户，4) 实时通信，5) 集成生态与 CI 质量体系。
- 结合现状建议定义里程碑：
  - **M1（可试点）**：真实驱动 + PostgreSQL + API 鉴权。
  - **M2（可团队级使用）**：WebSocket + 完整测试体系 + 报告模板化。
  - **M3（平台化）**：多租户 + 外部系统集成 + 审计合规。
